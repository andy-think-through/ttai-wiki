# Agent Browser

> Visual browser automation system using Claude's computer-use API to automate repetitive SaaS workflows with pixel-level interaction.
> Last updated: 2026-04-14

## What It Is

Agent Browser is a Node.js/TypeScript system that gives Claude Sonnet 4.6 control of a real Chromium browser via the Anthropic computer-use API. The agent sees screenshots, reasons about what's on screen, and sends mouse/keyboard actions at exact pixel coordinates. No DOM selectors, no web scraping APIs — pure visual interaction grounded in pixel-level observations.

The system operates as a core loop: capture screenshot → send to Claude with task description → receive action (click/type/key/scroll) → execute via Playwright → repeat until completion or bail-out.

## Architecture Direction (2026-04-13 discussion)

Andy raised a candidate architectural shift: **separate atomic actions from workflows**. Currently each workflow (schema, title/meta, keyword strategy) encodes its platform knowledge inline. Proposal is to make every UI interaction point in Hike a reusable action module (e.g., `updateTitleAction`, `updateMetaAction`, `pasteJsonLdAction`, `publishAction`), and compose workflows from those actions.

Practical next step agreed: **modular prompt fragments** — each action as a TypeScript function returning a prompt snippet, with its own coordinates, scroll behaviour, and gotchas (Ctrl+A vs triple-click, click-outside-before-scroll, etc.). Workflows compose fragments rather than duplicating per-field knowledge.

Reframes the Hike workflow discovery brief: from "what workflows should we build?" to "what actions does the agent need to know, and what workflows emerge from composing them?" Status: thinking captured, no code change yet.

Candidate new principle to lint: **capabilities before workflows** — map the platform's interaction surface first; workflows become cheap compositions afterwards.

## Architecture Highlights

### The Core Loop
1. Capture 1280×800 screenshot (downscaled to 832×520 for tokens)
2. Send screenshot + task to Claude
3. Claude responds with action (click, type, key, scroll)
4. Execute action via Playwright, capture new screenshot
5. Return as tool_result and repeat

### Cost Optimization (Critical)
- **Prompt caching (April 2026):** System prompt cached server-side via `prompt-caching-2024-07-31` beta. 1.25x write on iteration 1, 0.1x read on all subsequent. ~15-20% input token cost reduction per run. Three files changed (~30 lines total).
- **Screenshot downscaling:** Default 0.65x scale (~650 tokens) saves ~35% vs full-res
- **F6 text extraction:** ~100-500 tokens instead of full screenshots for data-heavy pages
- **Message compression:** Older screenshots replaced with text placeholders
- **Hard-coded coordinates:** Stable UI elements use pixel coordinates, no screenshot needed

Progression from naive to optimized: v1 Onsite Optimizer Phase 1 cost $2.50 → v3 $0.25 (10x reduction).

### Planned Optimisation Layers (April 2026 Research)

Research into Browser Use, Stagehand, Agent Browser (Vercel), Playwright MCP, and academic papers validated independently-discovered patterns and identified three optimisation layers:

1. **Prompt caching** — Done. ~15-20% input token reduction.
2. **Deterministic routing** — Next. Replace Onsite Optimizer Phase 2's 10-12 agent iterations with scripted Playwright sequence + single Claude API call for title/meta generation. Projected per-page cost from ~$0.72 to ~$0.10-0.15.
3. **Action caching** — Later. Stagehand's approach: first run uses LLM to find selectors, subsequent runs replay stored selector at zero LLM cost. ~80% speedup on repeat runs. Most relevant for KW Bell invoice processing where high repetition justifies infrastructure investment.

Industry context from research:
- **Browser Use 1.0:** Navigates primarily via DOM, screenshots only when needed; each screenshot adds ~0.8s latency
- **Agent Browser (Vercel):** 93% context reduction vs Playwright MCP via ASCII wireframe snapshots
- Both validate Andy's independently-discovered patterns: F6-first ≈ DOM-first; hardcoded coordinates ≈ manual action caching

### Bail-Out Controls
- Max iterations (default 25)
- Timeout (default 5 minutes)
- Stuck-loop detection (3 identical actions)
- Screenshot hash detection (3 unchanged screens)
- Domain allowlist (security boundary)
- Cost ceiling (default $2.00)

## Key Learnings

### The Navigation Rule
**Never navigate through dropdown menus.** Dropdowns fail constantly for pixel-based agents (hover states cause misfire). Solution: hardcode every URL and navigate via Ctrl+L + type + Enter.

### Screenshots Are the Most Expensive Thing
Cost hierarchy from cheapest to most expensive:
1. Hardcode known coordinates (0 tokens)
2. Hardcode known data into prompt (0 tokens)
3. F6 text extraction (~100-500 tokens)
4. Downscaled screenshot 0.65x (~650 tokens)
5. Full-resolution screenshot 1.0x (~1,000 tokens)

Good phases design for cost by using this hierarchy aggressively.

### Phase Decomposition Is Real Architecture
Each phase is a separate agent run with its own system prompt, iteration limit, cost ceiling, screenshot settings. Good boundaries separate concerns:
- **Research phases:** cheap (low screenshot scale, tight limits, F6-heavy)
- **Execution phases:** precise (high screenshot scale, generous limits)
- **Content generation:** direct API call (don't route through browser)

### Agent Recovery Resilience
Agents handle mid-run problems well: mid-phase ceiling hits, stuck-loop detection, modal dialogs, slow loads. Resilience comes from visual understanding. Not free though — each confused iteration costs tokens. Stuck-loop detectors exist because recovery instinct can become pathological.

### macOS Key Remapping (April 2026)
Computer-use API sends `ctrl+` shortcuts but macOS interprets Control differently from Cmd. Fixed via `normaliseKey` in `actions.ts` remapping `ctrl`/`control` to `Meta` (Cmd). Was silently causing select-all, copy/paste failures — agent recovered but burned iterations. Applies to any browser agent running on macOS.

### Trust the Agent, Not the DOM (April 2026)
Standalone Playwright scripts with Shadow DOM traversal failed after 2 hours on a Shopify SEO settings modal. The visual browser agent handled the same task in 14 iterations for $0.34. Pixel-based agents bypass Shadow DOM, React state, cross-origin iframes entirely. Principle: if a scripted approach fights the DOM, let the visual agent handle it instead.

### Textarea Interaction Rules (April 2026)
- Use Ctrl+A (Cmd+A via remap), not triple-click, for textarea selection — triple-click only selects one line/paragraph
- Click outside textarea before scrolling — scroll events inside a textarea scroll the field, not the page
- Save button position varies with variable-height fields (JSON-LD) — needs one screenshot to locate, unlike fixed-height fields where coordinates can be hardcoded

### Always Verify State Between Runs
Agent picked up stale customer URL from previous run's state file. New customer runs need explicit state reset. Applies to all workflow runs.

## Prompt Engineering for Visual Agents

- **Be prescriptive:** `key "ctrl+l"` + `type "url"` + `key "Return"` — not "navigate to page"
- **Tell it what NOT to do:** Explicit prohibitions save significant tokens
- **Front-load data:** Embed results from Phase 1 into Phase 2 prompt
- **Hardcode coordinates:** If testing confirms button always at (640, 266), embed that
- **Prefer F6 reads:** Use text extraction before screenshots
